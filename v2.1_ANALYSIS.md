# Douyin Skill v2.1 - 核心问题分析

## ❌ 为什么 v2.1 仍然失败？

### 问题根源

**我们的架构：**
```
浏览器登录 → 获取 Cookie → 关闭浏览器 → Node.js fetch API
```

**MediaCrawler 的架构：**
```
浏览器登录 → 保持浏览器打开 → 在浏览器内调用 API → 使用浏览器的完整上下文
```

### 关键差异

| 维度 | douyin_skill v2.1 | MediaCrawler |
|------|------------------|--------------|
| API 调用环境 | ❌ Node.js fetch | ✅ 浏览器 page.evaluate() |
| localStorage | ❌ 未使用 | ✅ 从浏览器提取 |
| User-Agent | ❌ 硬编码字符串 | ✅ page.evaluate("() => navigator.userAgent") |
| Cookie 同步 | ❌ 一次性提取 | ✅ 实时从 browser_context |
| 浏览器状态 | ❌ 关闭后调用 API | ✅ 保持打开状态 |
| playwright_page | ❌ 不传给 API | ✅ 传给 DouYinClient 使用 |

### 为什么这很重要？

抖音的风控检测：
1. **TLS 指纹** - Node.js 的 TLS 指纹和浏览器不同
2. **HTTP/2 指纹** - fetch 的实现和 Chrome 有差异
3. **请求顺序** - 真实浏览器会先加载页面，再调用 API
4. **localStorage 状态** - 抖音会检查特定的 localStorage 值
5. **Referer 一致性** - 浏览器内的请求有完整的 Referer 链

## ✅ 正确的解决方案

### 方案 1: 在浏览器内调用 API（推荐）

不使用 Node.js 的 fetch，改用：
```javascript
const response = await page.evaluate(async (url) => {
  const res = await fetch(url);
  return await res.json();
}, apiUrl);
```

**优势：**
- ✅ 使用浏览器的 TLS 指纹
- ✅ 使用浏览器的 HTTP/2 实现
- ✅ 自动带上所有浏览器状态
- ✅ localStorage、sessionStorage 都可用
- ✅ 完全真实的请求环境

### 方案 2: 完全迁移到 Python + Playwright（彻底）

直接使用 MediaCrawler 的架构：
```python
from playwright.async_api import async_playwright

# 打开浏览器
page = await context.new_page()
await page.goto("https://www.douyin.com")

# 在浏览器内调用 API
result = await page.evaluate("""
async () => {
  const response = await fetch('/aweme/v1/web/aweme/post/?...');
  return await response.json();
}
""")
```

### 方案 3: 使用 CDP 获取浏览器网络层（妥协）

通过 CDP 拦截浏览器的网络请求：
```javascript
await page.route('**/aweme/**', route => {
  // 让浏览器发送请求，我们拦截响应
  route.continue();
});
```

## 🔬 技术债务

### 为什么一开始没这么做？

1. **误以为 Cookie 足够** - 实际上还需要完整的浏览器上下文
2. **Node.js 更简单** - 但 TLS 指纹暴露了
3. **参考了 video-account-monitor** - 但它也有同样的问题

### 学到的教训

1. **爬虫不是拿到 Cookie 就行** - 需要完整的浏览器环境
2. **API 签名不是唯一的风控点** - TLS、HTTP/2 指纹同样重要
3. **stealth.min.js 只保护浏览器环境** - Node.js fetch 不受保护
